# 上下文重置与协议保留清单

日期：2026-06-29

## 重置决定

用户要求从当前轮次开始完全忘记此前已经实现过的研究内容。后续选题、方法设计、实验解释与论文叙事不得继续继承旧实现、旧方法名、旧实验结论或旧候选方向。

## 不再作为前提的内容

- 不再默认沿用此前任何方法实现、模块设计、训练框架或候选名称。
- 不再把旧实验中的正向/负向结果作为新研究问题的证据。
- 不再继续推进旧路线的补实验、调参、gate、selector、diagnostic 或 paper table。
- 旧目录和旧文档仅作为历史归档，除非用户明确要求查阅，否则不作为当前工作的上下文来源。

## 仍可保留的协议层内容

- 项目协作规则：中文沟通、只在当前工作目录内工作、任务结束同步更新 `AGENTS.md`、按 GitHub 仓库要求上传。
- 任务大方向：图学习 / 图自监督 / 图对比学习，主要任务仍可优先考虑节点分类；如用户另行指定，以新指定为准。
- 数据划分必须显式记录，不能把 `model_seed`、`split_index` 和 `split_seed` 混写。
- 小数据集评估应报告 split-level 波动，论文级实验优先使用多个 split，而不是只换模型随机种子。
- 同配图如果继续使用随机划分，应显式说明比例；此前确认过的 `train:val:test ~= 1:1:8` 只能作为可选协议模板，不代表新方法必须使用。
- 异配图如果使用 WebKB / WikipediaNetwork / Actor 等公开数据，应优先采用公开固定 split，并记录 mask 形状与 `split_index`。
- 主评估指标需要在新方向启动时重新确认；同一主表必须保持指标口径一致。
- 自监督训练不得使用测试标签；标签只允许在协议规定的 probe、validation、diagnostic 或最终评估中使用，并需明确写出用途。
- 每个 run 应保存完整 metadata：数据集、split 协议、`split_index`、`split_seed`、`model_seed`、命令、依赖版本、git commit/dirty 状态和关键超参数。
- baseline、新方法和消融必须在同一数据划分、同一评估入口和可比调参预算下比较。

## 后续工作原则

- 下一轮新 idea 需要先重新定义 research question、目标失败模式、baseline 清单和最小可发表证据，再决定是否写代码。
- 若需要新实现，应优先新建清晰隔离的工作区或明确重构入口，不默认复用旧方法代码。
- 如果复用旧代码中的通用工具，只能复用数据加载、split 生成、日志记录、汇总脚本等协议工具，并需要检查它们不夹带旧方法假设。
